# ReSale — C2C сервис объявлений

## 1. Тема и целевая аудитория

### 1.1 Тема проекта
**ReSale** — интернет-сервис для размещения объявлений и поиска товаров между частными лицами (C2C). Пользователи публикуют объявления, ищут товары по категориям и географии.

**Фокус MVP:** вертикаль **«Товары»** (электроника, дом/ремонт, хобби и т.п.).

### 1.2 Реальные аналоги (доказательство рыночной ниши)
- Avito (Россия)
- OLX (международный)
- Facebook Marketplace (международный)
- Craigslist (США)

### 1.3 Целевая аудитория (размер и местоположение)
**Местоположение:** Россия (все регионы).

**Размер аудитории для расчётов (ориентир — масштаб Avito):**
- **MAU:** **72M+** ежемесячная аудитория. [[1]](https://ads.avito.com/platform)
- **DAU:** **22 млн** ежедневная аудитория. [[2]](https://tass.ru/ekonomika/16064459)

Прогресс по количеству контента в сервисе:
- **220+ млн активных объявлений** по итогам апреля 2023 года. [[3]](https://moskva.mts.ru/about/media-centr/soobshheniya-kompanii/novosti-mts-v-rossii-i-mire/2024-12-17/mts-adtech-i-avito-obyavili-o-partnerstve-v-oblasti-razmeshheniya-nativnoj-reklamy?ysclid=mltp0fwznz612010775)
- **2+ млн новых объявлений в день**. [[3]](https://moskva.mts.ru/about/media-centr/soobshheniya-kompanii/novosti-mts-v-rossii-i-mire/2024-12-17/mts-adtech-i-avito-obyavili-o-partnerstve-v-oblasti-razmeshheniya-nativnoj-reklamy)
- **140 млн поисковых запросов в день**. [[4]](https://habr.com/ru/companies/avito/articles/429076/?ysclid=mltpwq0zb7242650464)
- **10 сделок в секунду**. [[5]](https://career.avito.com/)

### 1.4 MVP-функционал
1. **Выдача и поиск объявлений:** персональная лента (подборки/рекомендации), категории, фильтры (цена/состояние), сортировка, геолокация (город/район).
2. **Просмотр карточки объявления:** описание, цена, фото, локация, профиль продавца, кнопка «написать».
3. **Публикация объявления:** создание/редактирование/удаление объявлений, загрузка фото.
4. **Избранное:** добавление/удаление, список избранных.
5. **Жалобы и модерация:** жалоба на объявление/пользователя, скрытие/блокировка, очередь модерации.

### 1.5 Ключевые продуктовые решения
- **Геозависимая выдача, персонализированная лента обьявлений:** ранжирование с учётом города/расстояния, а также базовой системы рекомендаций.
- **Поиск как основной способ навигации по каталогу:** пользователь может формировать запрос и уточнять его с помощью фильтров (категория, цена, состояние, сортировка). Быстрый и точный поиск является ключевым сценарием использования сервиса.
- **Доверие к продавцу как часть продукта:** у продавца отображаются рейтинг и отметка о подтвержденном профиле, что повышает доверие покупателей и влияет на конверсию в контакт.
- **Модерация и защита от мошенничества обязательны:** автоматические правила и использование ИИ для автоматизации, спорные случаи отправляются на ручную модерацию.

#### Коэффициенты поведения (по выгрузке)

Формулы:
- Просмотры на один поисковый запрос: `V/Q = V_sample / Q_sample`
- Конверсия «просмотр → контакт»: `CR(V→C) = C_sample / V_sample`
- Контакты на один поисковый запрос: `C/Q = C_sample / Q_sample`

Результаты:
- `V/Q = 10 959 425 / 10 805 232 = 1,014`
- `CR(V→C) = 196 471 / 10 959 425 = 1,79%`
- `C/Q = 196 471 / 10 805 232 = 1,82%`

## 4. Расчёт локальной балансировки нагрузки

### 4.1 Схема балансировки нагрузки

![img_2.png](img_2.png)

На уровне **L4** используется **Cloud.ru Evolution Load Balancer**. Балансировщик работает на уровне **L4** и не выполняет **SSL/TLS termination**, поэтому завершение TLS выносится на уровень **L7**. [[15]](https://cloud.ru/products/evolution-load-balancer)

На уровне **L7** используется **NGINX ingress/reverse-proxy**, который выполняет:

- **SSL termination**;
- **HTTP/HTTPS-маршрутизацию** запросов;
- распределение трафика по backend-сервисам. [[16]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes)

Резервирование L7-групп принимается по схеме **N+1**:

`N_total = N_work + 1`

### 4.2 Расчёт количества балансировщиков

#### Сводная таблица расчёта L7

| Группа | Домен | Ограничение | Нод | CPU на ноду | Итого CPU |
|---|---|---|---:|---:|---:|
| Web | `www.resale.ru` | HTTPS CPS | 2 | 8 | 16 |
| Buyer API | `api.resale.ru` | сеть | 5 | 4 | 20 |
| Seller API | `seller.resale.ru` | сеть | 4 | 4 | 16 |
| **Итого** | — | — | **11** | — | **52** |

Для расчёта принимаются:

- `util = 0,8` — ограничение по загрузке ноды [[17]](https://www.npiontko.pro/2024/12/27/capacity-planning-utilization)
- `k_conn = 0,2` — коэффициент новых HTTPS-соединений, то есть  
  `CPS_peak = 0,2 * RPS_peak` [[19]](https://blog.cloudflare.com/measuring-network-connections-at-scale/)

По бенчмарку **NGINX Ingress Controller**:

- для профиля **4 CPU**  
  `BW_eff = 8,72 * 0,8 = 6,98 Гбит/с` [[16]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes)

- для профиля **8 CPU**  
  `BW_eff = 8,64 * 0,8 = 6,91 Гбит/с`  
  `CPS_eff = 3 399 * 0,8 = 2 719 conn/s`  
  [[16]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes) [[18]](https://blog.nginx.org/blog/testing-the-performance-of-nginx-and-nginx-plus-web-servers)

- для профиля **4 CPU** по HTTPS CPS  
  `CPS_eff = 1 735 * 0,8 = 1 388 conn/s` [[18]](https://blog.nginx.org/blog/testing-the-performance-of-nginx-and-nginx-plus-web-servers)

Профиль L7-ноды выбирается как минимальный по CPU, при котором сохраняется минимальное количество нод.

####  Группа `api.resale.ru`

Для `api.resale.ru`:

- `RPS_peak = 10 136,36`
- `CPS_peak = 0,2 * 10 136,36 = 2 027,27`
- `BW_peak = 24,81 Гбит/с`

Количество нод по сети:

`24,81 / 6,98 = 3,56 => 4 ноды`

Количество нод по новым HTTPS-соединениям:

`2 027,27 / 1 388 = 1,46 => 2 ноды`

Принимается:

- `N_work = max(4, 2) = 4`
- `N_total = 4 + 1 = 5`

Выбранный профиль:

- **5 нод по 4 CPU**
- `CPU_total = 5 * 4 = 20`


#### Группа `seller.resale.ru`

Для `seller.resale.ru`:

- `RPS_peak = 555,55`
- `CPS_peak = 0,2 * 555,55 = 111,11`
- `BW_peak = 16,32 Гбит/с`

Количество нод по сети:

`16,32 / 6,98 = 2,34 => 3 ноды`

Количество нод по новым HTTPS-соединениям:

`111,11 / 1 388 = 0,08 => 1 нода`

Принимается:

- `N_work = max(3, 1) = 3`
- `N_total = 3 + 1 = 4`

Выбранный профиль:

- **4 ноды по 4 CPU**
- `CPU_total = 4 * 4 = 16`

#### Группа `www.resale.ru`

Для `www.resale.ru` берётся пиковый `RPS` всей системы из раздела 2:

`RPS_peak,total = 10 691,91`

Пиковая нагрузка по новым HTTPS-соединениям:

`CPS_peak,www = 0,2 * 10 691,91 = 2 138,38`

Количество нод по новым HTTPS-соединениям для профиля **8 CPU**:

`2 138,38 / 2 719 = 0,79 => 1 нода`

Принимается:

- `N_work = 1`
- `N_total = 1 + 1 = 2`

Выбранный профиль:

- **2 ноды по 8 CPU**
- `CPU_total = 2 * 8 = 16`

## 5. Логическая схема БД

- факт запроса контакта хранится в `event_log`;
- состояние модерации хранится в `complaints`;

### 5.1 Описание таблиц

| Таблица               | Описание                                                                                                                                        |
|-----------------------|-------------------------------------------------------------------------------------------------------------------------------------------------|
| `users`               | Данные пользователя и публичные данные продавца: имя, рейтинг, верификация, URL аватарки, статус                                                |
| `categories`          | Иерархическая структура категорий объявлений. Поле `attributes_schema_json` хранит схему специфичных атрибутов категории                        |
| `locations`           | Иерархическая структура географии: страна, регион, город, район                                                                                 |
| `cards`               | Объявления пользователей. Поле `image_urls_json` хранит массив URL изображений, чтобы не выполнять join при выдаче карточки                     |
| `storage`             | Файловые данные: оригиналы фото и thumbnail-версии. Используется для хранения изображений объявлений и аватарок                                 |
| `favorites`           | Текущее состояние избранного пользователя                                                                                                       |
| `complaints`          | Жалобы на пользователя или объявление. Содержит также поля модерации: статус, приоритет, модератор, время решения                               |
| `event_log`           | Общий журнал действий пользователей. Сюда записываются поиск, просмотр карточки, запрос контакта, изменение объявления, жалоба и другие события |
| `card_search`         | Денормализованная поисковая проекция объявления. Обновляется асинхронно после изменения `cards`                                                 |
| `search_result_cache` | Кэш результатов поисковой выдачи по запросу, фильтрам и географии                                                                               |
| `card_cache`          | Кэш сериализованной карточки объявления                                                                                                         |

### 5.2 Размеры данных и нагрузки на чтение/запись

Значения `RPS` на чтение и запись получены из действий, рассчитанных в разделе 2:

- поиск/выдача — `1620,37 RPS`;
- просмотр карточки — `1643,49 RPS`;
- запрос контакта — `29,46 RPS`;
- избранное — `82,17 RPS`;
- жалоба — `3,29 RPS`;
- публикация объявления (CRUD) — `23,15 RPS`;
- загрузка фото — `162,04 RPS`.

Для таблицы `users` прирост рассчитан по увеличению пользовательской аудитории на **11 млн за 2 года**:

- `Users_day = 11 000 000 / 730 = 15 068,49`
- `RPS_write(users) = 15 068,49 / 86 400 = 0,1744`
- при размере записи `256 Б`:
- `Growth_day(users) = 15 068,49 * 256 = 3,86 МБ/сутки`

Для `categories` и `locations` запись принимается равной `0 RPS`, так как это справочники, а их изменение не входит в основной продуктовый поток нагрузки.

Для таблиц-кэшей значение в колонке **«Прирост данных за сутки»** - уточный объём перезаписи.

| Таблица               | Объем записи, байт | Прирост данных за сутки, ГБ | Нагрузка на запись, RPS | Нагрузка на чтение, RPS |
|-----------------------|-------------------:|----------------------------:|------------------------:|------------------------:|
| `users`               |                256 |      `0,00386 ГБ (3,86 МБ)` |                  0,1744 |                 1672,95 |
| `categories`          |                224 |                        0,00 |                    0,00 |                 3287,01 |
| `locations`           |                128 |                        0,00 |                    0,00 |                 3287,01 |
| `cards`               |               2048 |                        4,10 |                   23,15 |                 1699,39 |
| `storage`             |          4 329 472 |                   60 613,72 |                  162,04 |                 3263,86 |
| `favorites`           |                 32 |                       0,227 |                   82,17 |                 1643,49 |
| `complaints`          |                256 |                      0,0728 |                    3,29 |                    3,29 |
| `event_log`           |                128 |                       39,41 |                 3563,97 |                 3563,97 |
| `card_search`         |               1024 |                        2,05 |                   23,15 |                 1620,37 |
| `search_result_cache` |               1024 |                       28,67 |                  324,07 |                 1620,37 |
| `card_cache`          |               2048 |                       33,18 |                  187,50 |                 1643,49 |

Принятые размеры записи:

- `cards` — **2 КБ**, как и размер метаданных объявления в разделе 2;
- `storage` — **4 МБ + 132 КБ**, то есть оригинал фото и уменьшенная копия;
- `favorites` — **32 Б**, как и в разделе 2;
- `complaints` — **256 Б**, как и в разделе 2;
- `event_log`, `card_search`, `search_result_cache`, `card_cache` — логические размеры записей по составу полей.

Принятые правила распределения `RPS` по таблицам:

- `users`: чтение = получение продавца в карточке и связанных пользовательских данных; запись = прирост пользовательской базы;
- `categories`: чтение = поиск + просмотр карточки + CRUD; запись = `0 RPS`, так как это справочник;
- `locations`: чтение = поиск + просмотр карточки + CRUD; запись = `0 RPS`, так как это справочник;
- `cards`: чтение = просмотр карточки + запрос контакта + жалоба + CRUD;
- `storage`: запись = загрузка фото, чтение = поиск + просмотр карточки;
- `event_log`: запись и чтение = сумма всех пользовательских действий;
- `card_search`: чтение = поиск, запись = CRUD;
- `search_result_cache`: запись = `20%` от поиска;
- `card_cache`: запись = `10%` от просмотров карточки + `CRUD`.

### 5.3 Требования к консистентности

| Таблица               | Требование к консистентности                                      | Критерий к запросам                                                                                                                                  | Описание                                                                                        |
|-----------------------|-------------------------------------------------------------------|------------------------------------------------------------------------------------------------------------------------------------------------------|-------------------------------------------------------------------------------------------------|
| `users`               | strong consistency                                                | запись допускается только для существующего или корректно создаваемого пользователя после проверки уникальности идентификаторов и обязательных полей | Статус пользователя, верификация и публичные поля продавца должны читаться однозначно           |
| `categories`          | strong consistency                                                | запись допускается только для административного изменения справочника при существующем `parent_id` либо при создании корневой категории              | Это справочник, изменения редкие                                                                |
| `locations`           | strong consistency                                                | запись допускается только для административного изменения справочника при существующем `parent_id` либо при создании корневой локации                | Это справочник, изменения редкие                                                                |
| `cards`               | strong consistency                                                | запись допускается только если продавец существует, категория и локация валидны, а изменение выполняет владелец объявления или уполномоченный сервис | После CRUD продавец должен сразу видеть актуальное состояние объявления                         |
| `storage`             | strong consistency для метаданных, read-after-write для владельца | запись допускается только если существует владелец файла, а при наличии `card_id` объявление существует и доступно владельцу                         | После загрузки фото продавец должен сразу видеть загруженное изображение                        |
| `favorites`           | strong consistency                                                | запись допускается только если существуют пользователь и объявление, а пара (`user_id`, `card_id`) уникальна                                         | Добавление и удаление из избранного должно сразу отражаться в интерфейсе                        |
| `complaints`          | strong consistency                                                | запись допускается только если существует автор жалобы и указан хотя бы один объект жалобы: `target_user_id` или `target_card_id`                    | Жалоба не должна теряться, её статус должен меняться однозначно                                 |
| `event_log`           | at-least-once append                                              | запись допускается только как append-only событие с существующим `user_id`; связанные сущности, если переданы, должны быть валидны                   | Событие действия пользователя не должно теряться, допускается идемпотентная повторная обработка |
| `card_search`         | eventual consistency                                              | запись допускается только как результат асинхронного обновления после успешной фиксации изменений в `cards`                                          | Поисковая проекция обновляется асинхронно после изменения объявления                            |
| `search_result_cache` | eventual consistency / TTL                                        | запись допускается только для нормализованного ключа запроса, сформированного из `query + filters + location`                                        | Допустима кратковременная устарелость поисковой выдачи                                          |
| `card_cache`          | eventual consistency / TTL                                        | запись допускается только для существующего объявления и должна инвалидироваться после изменения `cards`                                             | Допустима кратковременная устарелость карточки вне seller flow                                  |

### 5.4 Особенности распределения

| Таблица               | Основные ключи доступа                            | Характер распределения  | Описание                                                                                                        |
|-----------------------|---------------------------------------------------|-------------------------|-----------------------------------------------------------------------------------------------------------------|
| `users`               | `id`                                              | близкое к равномерному  | Основной доступ идёт по идентификатору пользователя                                                             |
| `categories`          | `id`, `parent_id`                                 | неравномерное           | Небольшое число верхнеуровневых категорий читается чаще остальных                                               |
| `locations`           | `id`, `parent_id`                                 | неравномерное           | Крупные города и регионы дают существенно больше запросов, чем малые населённые пункты                          |
| `cards`               | `id`, `seller_id`, `category_id`, `location_id`   | выраженно неравномерное | Популярные объявления, крупные продавцы, крупные города и популярные категории формируют основную долю нагрузки |
| `storage`             | `card_id`, `storage_key`                          | выраженно неравномерное | Горячие ключи возникают на популярных объявлениях и их изображениях                                             |
| `favorites`           | `user_id`, `card_id`                              | неравномерное           | Запись в основном распределяется по `user_id`, чтение и агрегаты концентрируются на популярных `card_id`        |
| `complaints`          | `target_card_id`, `target_user_id`                | неравномерное           | Жалобы концентрируются на проблемных объявлениях и подозрительных пользователях                                 |
| `event_log`           | `user_id`, `card_id`, `action_type`               | волнообразное           | Нагрузка идёт всплесками, синхронно с пользовательскими действиями                                              |
| `card_search`         | `category_id`, `location_id`, полнотекстовый ключ | выраженно неравномерное | Основной поток чтений сосредоточен в популярных категориях и крупных городах                                    |
| `search_result_cache` | `query + filters + location + page`               | сильно неравномерное    | Небольшое число популярных запросов и геосценариев даёт большую долю попаданий в кэш                            |
| `card_cache`          | `card_id`                                         | выраженно неравномерное | Карточки популярных объявлений читаются значительно чаще среднего                                               |
