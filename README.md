# ReSale — C2C сервис объявлений

## 1. Тема и целевая аудитория

### 1.1 Тема проекта
**ReSale** — интернет-сервис для размещения объявлений и поиска товаров между частными лицами (C2C). Пользователи публикуют объявления, ищут товары по категориям и географии.

**Фокус MVP:** вертикаль **«Товары»** (электроника, дом/ремонт, хобби и т.п.).

### 1.2 Реальные аналоги (доказательство рыночной ниши)
- Avito (Россия)
- OLX (международный)
- Facebook Marketplace (международный)
- Craigslist (США)

### 1.3 Целевая аудитория (размер и местоположение)
**Местоположение:** Россия (все регионы).

**Размер аудитории для расчётов (ориентир — масштаб Avito):**
- **MAU:** **72M+** ежемесячная аудитория. [[1]](https://ads.avito.com/platform)
- **DAU:** **22 млн** ежедневная аудитория. [[2]](https://tass.ru/ekonomika/16064459)

Прогресс по количеству контента в сервисе:
- **220+ млн активных объявлений** по итогам апреля 2023 года. [[3]](https://moskva.mts.ru/about/media-centr/soobshheniya-kompanii/novosti-mts-v-rossii-i-mire/2024-12-17/mts-adtech-i-avito-obyavili-o-partnerstve-v-oblasti-razmeshheniya-nativnoj-reklamy?ysclid=mltp0fwznz612010775)
- **2+ млн новых объявлений в день**. [[3]](https://moskva.mts.ru/about/media-centr/soobshheniya-kompanii/novosti-mts-v-rossii-i-mire/2024-12-17/mts-adtech-i-avito-obyavili-o-partnerstve-v-oblasti-razmeshheniya-nativnoj-reklamy)
- **140 млн поисковых запросов в день**. [[4]](https://habr.com/ru/companies/avito/articles/429076/?ysclid=mltpwq0zb7242650464)
- **10 сделок в секунду**. [[5]](https://career.avito.com/)

### 1.4 MVP-функционал
1. **Выдача и поиск объявлений:** персональная лента (подборки/рекомендации), категории, фильтры (цена/состояние), сортировка, геолокация (город/район).
2. **Просмотр карточки объявления:** описание, цена, фото, локация, профиль продавца, кнопка «написать».
3. **Публикация объявления:** создание/редактирование/удаление объявлений, загрузка фото.
4. **Избранное:** добавление/удаление, список избранных.
5. **Жалобы и модерация:** жалоба на объявление/пользователя, скрытие/блокировка, очередь модерации.

### 1.5 Ключевые продуктовые решения
- **Геозависимая выдача, персонализированная лента обьявлений:** ранжирование с учётом города/расстояния, а также базовой системы рекомендаций.
- **Поиск как основной способ навигации по каталогу:** пользователь может формировать запрос и уточнять его с помощью фильтров (категория, цена, состояние, сортировка). Быстрый и точный поиск является ключевым сценарием использования сервиса.
- **Доверие к продавцу как часть продукта:** у продавца отображаются рейтинг и отметка о подтвержденном профиле, что повышает доверие покупателей и влияет на конверсию в контакт.
- **Модерация и защита от мошенничества обязательны:** автоматические правила и использование ИИ для автоматизации, спорные случаи отправляются на ручную модерацию.

#### Коэффициенты поведения (по выгрузке)

Формулы:
- Просмотры на один поисковый запрос: `V/Q = V_sample / Q_sample`
- Конверсия «просмотр → контакт»: `CR(V→C) = C_sample / V_sample`

Результаты:
- `V/Q = 10 959 425 / 10 805 232 = 1,014`
- `CR(V→C) = 196 471 / 10 959 425 = 1,79%`
- `C/Q = 196 471 / 10 805 232 = 1,82%`

#### 2.2.4 Нагрузка на алгоритмические сервисы

После выбора алгоритмов поиска и рекомендаций появляется дополнительная внутренняя нагрузка.

В текущей версии расчёта численно оцениваются только те внутренние контуры, для которых база расчёта уже зафиксирована в разделе 2.

Формулы:

`RPS_avg = N_day / 86 400`

`RPS_peak = 3 * RPS_avg`

Для `feature-updater` входной поток считается как сумма событий, используемых в алгоритмах поиска и рекомендаций:

`Events_feature_day = Q_day + V_day + Fav_day + C_day`

Расчёт:

`Events_feature_day = 140 000 000 + 141 997 830 + 7 099 892 + 2 545 613 = 291 643 335`

`RPS_avg(feature-updater) = 291 643 335 / 86 400 = 3 375,50`

`RPS_peak(feature-updater) = 3 * 3 375,50 = 10 126,50`

## 4. Расчёт локальной балансировки нагрузки

### 4.1 Схема балансировки нагрузки

![img_2.png](img_2.png)

На уровне **L4** используется **Cloud.ru Evolution Load Balancer**. Балансировщик работает на уровне **L4** и не выполняет **SSL/TLS termination**, поэтому завершение TLS выносится на уровень **L7**. [[15]](https://cloud.ru/products/evolution-load-balancer)

На уровне **L7** используется **NGINX ingress/reverse-proxy**, который выполняет:

- **SSL termination**;
- **HTTP/HTTPS-маршрутизацию** запросов;
- распределение трафика по backend-сервисам. [[16]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes)

Резервирование L7-групп принимается по схеме **N+1**:

`N_total = N_work + 1`

  `CPS_peak = 0,2 * RPS_peak` [[19]](https://blog.cloudflare.com/measuring-network-connections-at-scale/)

По бенчмарку **NGINX Ingress Controller**:

- для профиля **4 CPU**  
  `BW_eff = 8,72 * 0,8 = 6,98 Гбит/с` [[16]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes)

- для профиля **8 CPU**  
  `BW_eff = 8,64 * 0,8 = 6,91 Гбит/с`  
  `CPS_eff = 3 399 * 0,8 = 2 719 conn/s`  
  [[16]](https://blog.nginx.org/blog/testing-performance-nginx-ingress-controller-kubernetes) [[18]](https://blog.nginx.org/blog/testing-the-performance-of-nginx-and-nginx-plus-web-servers)

- для профиля **4 CPU** по HTTPS CPS  
  `CPS_eff = 1 735 * 0,8 = 1 388 conn/s` [[18]](https://blog.nginx.org/blog/testing-the-performance-of-nginx-and-nginx-plus-web-servers)

Профиль L7-ноды выбирается как минимальный по CPU, при котором сохраняется минимальное количество нод.

####  Группа `api.resale.ru`

Для `api.resale.ru`:

- `RPS_peak = 10 136,36`
- `CPS_peak = 0,2 * 10 136,36 = 2 027,27`
- `BW_peak = 24,81 Гбит/с`

Количество нод по сети:

`24,81 / 6,98 = 3,56 => 4 ноды`

Количество нод по новым HTTPS-соединениям:

`2 027,27 / 1 388 = 1,46 => 2 ноды`

Принимается:

- `N_work = max(4, 2) = 4`
- `N_total = 4 + 1 = 5`

Выбранный профиль:

- **5 нод по 4 CPU**
- `CPU_total = 5 * 4 = 20`


#### Группа `seller.resale.ru`

Для `seller.resale.ru`:

- `RPS_peak = 555,55`
- `CPS_peak = 0,2 * 555,55 = 111,11`
- `BW_peak = 16,32 Гбит/с`

Количество нод по сети:

`16,32 / 6,98 = 2,34 => 3 ноды`

Количество нод по новым HTTPS-соединениям:

`111,11 / 1 388 = 0,08 => 1 нода`

Принимается:

- `N_work = max(3, 1) = 3`
- `N_total = 3 + 1 = 4`

Выбранный профиль:

- **4 ноды по 4 CPU**
- `CPU_total = 4 * 4 = 16`

#### Группа `www.resale.ru`

Для `www.resale.ru` берётся пиковый `RPS` всей системы из раздела 2:

`RPS_peak,total = 10 691,91`

Пиковая нагрузка по новым HTTPS-соединениям:

`CPS_peak,www = 0,2 * 10 691,91 = 2 138,38`

Количество нод по новым HTTPS-соединениям для профиля **8 CPU**:

`2 138,38 / 2 719 = 0,79 => 1 нода`

Принимается:

- `N_work = 1`
- `N_total = 1 + 1 = 2`

Выбранный профиль:

- **2 ноды по 8 CPU**
- `CPU_total = 2 * 8 = 16`

### 6.4 Схема документа `card_search` в OpenSearch

Размер поисковой проекции считается по числу объявлений и среднему размеру документа из раздела 5:

- `N_cards = 220 000 000`
- `S_doc = 1024 Б`

Формула:

`S_card_search = N_cards * S_doc`

Расчёт:

`S_card_search = 220 000 000 * 1024 Б = 225 280 000 000 Б = 209,8 GiB ≈ 225,28 ГБ`

Для search-heavy нагрузки рекомендуется держать размер primary shard в диапазоне **10–30 GiB**.[[23]](https://docs.aws.amazon.com/opensearch-service/latest/developerguide/bp-sharding.html).

Принимается:

- `primary shards = 8`
- `replica shards = 1`

Проверка:

`209,8 GiB / 8 = 26,2 GiB на один primary shard`

### 6.6 Кеши и буферы

#### 6.6.1 Кеши

Используется паттерн **cache-aside** [[24]](https://learn.microsoft.com/en-us/azure/architecture/patterns/cache-aside)

Метрика cache hit ratio считается по формуле Redis:

`CHR = cache_hits / (cache_hits + cache_misses)` [[25]](https://redis.io/docs/latest/operate/rs/monitoring/get-started/)

Целевые диапазоны CHR:

- `> 90%` — excellent
- `70–90%` — good
- `< 70%` — needs attention [[26]](https://redis.io/tutorials/operate/observability/redis-software-prometheus-and-grafana/) [[27]](https://redis.io/tutorials/redis-software-observability-playbook/)

Пиковые чтения для кэшей считаются так:

- для `categories cache`, `locations cache`, `favorites cache`  
  `RPS_read_peak = RPS_read * k_peak`
- для `cards cache`  
  `RPS_read_peak = RPS_peak(card view) = 4 930,48` - кэшируется именно готовый ответ API просмотра карточки

Формулы:

`CHR_calc ≈ 1 - RPS_write_peak / RPS_read_peak`

`N_hot_keys = RPS_read_peak * TTL`

`S_hotset = N_hot_keys * S_record`

## 7. Алгоритмы

Для ReSale такими алгоритмами являются:

1. поиск объявлений;
2. рекомендательная лента объявлений.

### 7.1 Итоговые принятые решения

1. Поиск реализуется как многоэтапный pipeline:
   `разбор запроса -> отбор кандидатов -> ранжирование -> постобработка`. [[33]](https://habr.com/ru/companies/avito/articles/846832/) [[36]](https://habr.com/ru/companies/ozontech/articles/667600/)

2. Базовый поиск  — `lexical-first search` на OpenSearch.
  [[37]](https://docs.opensearch.org/latest/vector-search/ai-search/hybrid-search/index/) 

3. Рекомендательная лента на запуске реализуется как `heuristic + content-based recommendation` на основе `event_log`, `favorites`, `cards` и `card_search`.
   Применяется на раннем этапе. [[34]](https://habr.com/ru/companies/avito/articles/338470/) [[35]](https://habr.com/ru/companies/avito/articles/1004694/)

4. Основной алгоритм рекомендаций реализуется как `candidate generation -> ranking`, где retrieval и финальное ранжирование разделены.
   Для retrieval используется `two-tower` схема, а векторный поиск выполняется внутри OpenSearch. [[35]](https://habr.com/ru/companies/avito/articles/1004694/) [[39]](https://docs.opensearch.org/latest/vector-search/) [[40]](https://docs.opensearch.org/latest/vector-search/getting-started/concepts/)

### 7.2 Алгоритм поиска объявлений

#### 7.2.1 Блок применения и постановка задачи

Блок применения: buyer-path, endpoint поиска и выдачи объявлений.

Задача: по текстовому запросу пользователя, фильтрам и географическому контексту вернуть top-N активных объявлений, релевантных запросу, с учётом категории, цены, локации, свежести и доверия к продавцу.

Поиск выполняется по отдельной поисковой проекции `card_search` в OpenSearch.

#### 7.2.2 Используемые данные и структуры

| Структура                            | Назначение                                |
| ------------------------------------ | ----------------------------------------- |
| `card_search`                        | основная поисковая проекция               |
| `categories`, `locations`            | справочники для фильтров                  |
| `users`                              | рейтинг и верификация продавца            |
| `event_log`                          | поисковые запросы и поведенческие сигналы |
| Redis                                | кэши горячих карточек и справочников      |
| Kafka `cards-upsert`, `cards-delete` | обновление поискового индекса             |

#### 7.2.3 Последовательность работы алгоритма поиска
1. Нормализация и токенизация запроса.
2. Выделение из текста категории, бренда, модели и других фильтров, если они выражены явно.
3. Отбор кандидатов в OpenSearch по обратному индексу.
4. Применение обязательных фильтров: `status = active`, категория, локация, диапазон цены, фильтры по `attributes_json`.
5. Ранжирование найденных кандидатов по лёгкой формуле.
6. Переранжирование top-K документов с учётом дополнительных факторов.
7. Постобработка: удаление дубликатов, ограничение числа объявлений одного продавца, пагинация.
8. Логирование поискового события в `event_log`.

#### 7.2.4 Формула ранжирования

Для MVP принимается формула:

`Score_search(q, d, u) = w1 · BM25(q, d) + w2 · GeoScore(u, d) + w3 · Freshness(d) + w4 · SellerTrust(d)`

где:

1. `BM25(q, d)` — текстовая релевантность;
2. `GeoScore(u, d)` — близость объявления к пользователю;
3. `Freshness(d)` — повышение веса новых объявлений;
4. `SellerTrust(d)` — доверие к продавцу: рейтинг и верификация.

На финальном шаге к выдаче дополнительно применяются правила постобработки:

1. удаление слишком похожих карточек;
2. ограничение количества объявлений одного продавца;
3. ограничение чрезмерного доминирования одной категории в первых позициях.

#### 7.2.5 Проблемы поиска и способы решения

| Проблема                        | Решение                                                                                                                                                   |
| ------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Опечатки в запросе              | нечёткий поиск и исправление опечаток [[38]](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/search-workloads.html)     |
| Разные названия одного товара   | словарь синонимов и нормализация терминов [[38]](https://docs.aws.amazon.com/wellarchitected/latest/amazon-opensearch-service-lens/search-workloads.html) |
| Слишком узкая или пустая выдача | ослабление слишком жёстких условий запроса и переход к похожим объявлениям                                                                                |
| Геозависимость выдачи           | фильтр по `location_id` и географический коэффициент в ранжировании                                                                                       |
| Новые объявления без истории    | повышение веса новых объявлений и учёт доверия к продавцу                                                                                                 |
| Однообразная выдача             | удаление повторяющихся карточек и ограничение числа объявлений одного продавца                                                                            |

### 7.3 Алгоритм рекомендательной ленты объявлений

#### 7.3.1 Блок применения и постановка задачи

Блок применения:

1. главная лента;
2. блок «похожие объявления»;
3. рекомендации после просмотра карточки.

Задача: по истории действий пользователя и контексту текущей сессии построить ранжированный список объявлений, учитывающий интересы пользователя, географию, свежесть объявлений и доверие к продавцу.

#### 7.3.2 Используемые данные

| Структура              | Назначение                                                     |
| ---------------------- | -------------------------------------------------------------- |
| `event_log`            | история поисковых запросов, просмотров, избранного и контактов |
| `favorites`            | сильный сигнал интереса                                        |
| `cards`, `card_search` | источник кандидатов                                            |
| `users`                | seller trust                                                   |
| Kafka `user-actions`   | доставка событий в recommendation pipeline                     |
| Redis                  | кэш горячих данных ленты                                       |

#### 7.3.3 Последовательность работы на запуске(cold start)

На этапе запуска используется `heuristic + content-based recommendation`.

1. Из `event_log` выбираются последние действия пользователя: поисковые запросы, просмотры карточек, избранное, запросы контакта.
2. Более новые действия получают больший вес.
3. По истории строится профиль пользователя: интересующие категории, интересующий диапазон цен, интересующая география.
4. Генерируются кандидаты из нескольких источников: популярные в регионе, свежие в регионе, похожие на просмотренные, похожие на избранные.
5. На объединённом наборе кандидатов рассчитывается итоговый score.
6. Применяется постобработка: ограничение однообразия ленты, ограничение числа карточек одного продавца, исключение уже просмотренных карточек в текущей сессии.
7. Возвращается итоговый top-N.

#### 7.3.4 Формула скоринга рекомендаций

Для этапа запуска принимается формула:

`Score_rec(u, d) = a1 · ContentSim(u, d) + a2 · GeoPopularity(d) + a3 · Freshness(d) + a4 · SellerTrust(d) + a5 · PriceAffinity(u, d) - a6 · DiversityPenalty(d, L)`

где:

1. `ContentSim(u, d)` — похожесть объявления на профиль интересов пользователя;
2. `GeoPopularity(d)` — популярность объявления в регионе;
3. `Freshness(d)` — повышение веса новых объявлений;
4. `SellerTrust(d)` — доверие к продавцу;
5. `PriceAffinity(u, d)` — близость цены к привычному диапазону пользователя;
6. `DiversityPenalty(d, L)` — штраф за слишком похожие объявления в текущей ленте `L`.

#### 7.3.5 Проблемы рекомендаций и способы решения

| Проблема                                    | Решение                                                                      |
| ------------------------------------------- | ---------------------------------------------------------------------------- |
| Новый пользователь без истории              | региональная популярность и свежесть                                         |
| Новое объявление без статистики             | контентная похожесть и повышение веса новых объявлений                       |
| Быстрая смена интересов                     | повышенный вес у последних действий и поисковых запросов                     |
| Однообразная лента                          | ограничение однотипных карточек и ограничение числа карточек одного продавца |
| Доминирование слишком популярных объявлений | ограничение влияния популярности в итоговом score                            |
| Недоверие к части объявлений                | учёт модерации, рейтинга и верификации продавца                              |

#### 7.3.6 Зрелый этап развития

После накопления достаточной истории рекомендационная система переходит от простого content-based ранжирования к двухэтапной схеме:

`отбор кандидатов -> финальное ранжирование`

На этом этапе используется `two-tower` рекомендательная модель.

Первая башня строит векторное представление пользователя по его истории действий:

1. поисковые запросы;
2. просмотры;
3. избранное;
4. контакты;
5. сессионный контекст.

Вторая башня строит векторное представление объявления по его признакам:

1. категория;
2. цена;
3. география;
4. текст карточки;
5. seller trust;
6. поведенческие признаки объявления.

Формально:

`e_u = UserTower(history_u)`

`e_d = ItemTower(features_d)`

`Score_retrieval(u, d) = <e_u, e_d>`

где:

1. `e_u` — вектор пользователя;
2. `e_d` — вектор объявления;
3. `<e_u, e_d>` — скалярное произведение, которое определяет близость пользователя и объявления.

Последовательность действий алгоритма:

1. По истории пользователя строится пользовательский вектор.
2. Полученный пользовательский вектор хранится как короткоживущие производные данные в Redis и используется `recommendation-service` в online path.
3. По каталогу объявлений заранее строятся векторы карточек.
4. Векторы карточек хранятся в OpenSearch в поле `item_embedding`.
5. По близости векторов выполняется быстрый отбор кандидатов.
6. На найденном наборе кандидатов выполняется отдельное финальное ранжирование.

Итоговый score:

`Score_rec_final(u, d) = b1 · RetrievalScore(u, d) + b2 · Freshness(d) + b3 · SellerTrust(d) + b4 · PriceAffinity(u, d) - b5 · DiversityPenalty(d, L)`

Смысл данного этапа в том, что система уже не просто сравнивает карточки по заданным правилам, а  учитывает скрытые закономерности поведения пользователя.
